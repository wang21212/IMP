# IMP Intent — 意图三循环（描述 → 细化 → 可行性）

## 触发条件

由 Global Rules 路由：功能级或骨架级工作开始时

## 三个阶段

### Phase 1: 描述（捕获原始意图）

让用户/AI 把当前意图用最自然的方式说出来。
AI 此时不纠正、不补充，只记录，输出：

```
[Intent-描述] {原始描述文本}
```

### Phase 2: 细化（多轮澄清，最多 5 问）

AI 基于描述，以 AI 对此 intent 的认知和理解，识别模糊点，可以包含但不限于：

- 用户入口是什么？（从哪里触发）
- 预期结果是什么？（完成后看到什么）
- 边界条件是什么？（什么情况不在范围内）
- 与现有功能的关系？（新建/替换/扩展）
- 优先级与约束？（必须支持什么，不能破坏什么）

每轮问完等用户回答，不得一次问全部。

**中途落盘**：若细化过程超过3轮对话仍未结束，AI 应将当前已确认的意图点追加到 `intent-log.md`，标注 `[draft]`。目的是防止长时间讨论导致上下文窗口丢失早期意图。锁定后用最终版本覆盖草稿条目。

细化完成后输出：

```
[Intent-细化] {结构化意图摘要，含用户入口/预期结果/边界/关系/约束}
```

### Phase 3: 可行性验证（三维度检查）

AI 先 Reasoning（不读文件）：

```
[Intent-可行性-Reasoning]
横向冲突：{与已知 Spec/功能是否冲突？基于上下文推断}
纵向兼容：{现有骨架是否能支撑？基于已知架构推断}
意图完备度：{任务复杂度 vs 当前意图描述量是否匹配？轻量/中等/复杂}
不确定点：{列出需要查证的点（若有）}
```

若存在不确定点 → 主动扫描相关文件（direction.md、相关代码）：

```
[Intent-可行性-证据] 读取 {文件名}，确认：{结论}
```

最终输出三维度结论：

```
[Intent-可行性] 横向: ✅/⚠️/❌  纵向: ✅/⚠️/❌  完备度: ✅匹配/⚠️偏少/❌远不足
```

- 若横向或纵向 ⚠️/❌ → 回到 Phase 2 重新细化，不得继续执行
- 若完备度 ⚠️/❌ → 不直接锁定，继续多轮问答深化意图，直到完备度匹配（复杂任务意图深化后，IMP 可作为 SOP 产出物生成需求文档，放项目 docs/ 目录，这是 IMP 过程的产出物，不是内部 memory 状态）
- 若三维度均 ✅ → 输出锁定意图，进入对应执行 Skill

```
[Intent-锁定] {最终意图摘要，作为后续执行的输入}
```

### Phase 3 完成后：写入 intent-log（任务级可跳过）

追加一条到 `.windsurf/memory/intent-log.md`。

写入指令：

- 用你认为足够还原本次意图的详略，记录这次锁定的内容
- 你判断什么信息未来最可能被需要，就记什么，不规定格式和字数
- 必须包含三个最小字段：日期、级别（功能级/骨架级）、一句话摘要
- 完备度评估结果也应记入：复杂任务记详细意图，简单任务记简要意图，确保 log 里的详略程度与任务实际复杂度自然匹配
- 其余由你自己决定记多还是记少
