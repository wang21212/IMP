# IMP Feature — 功能迭代流程

## 适用范围

N→N.1 迭代，骨架不动，新增或修改一个功能单元。
（进入此 Skill 前已完成 imp-intent，意图已锁定）

## 流程

### Step 1: 读取项目上下文 + 对当前里程碑重新判级

读 `direction.md`（目标与约束）、`milestone-state.md`（只读最新版本快照——文件顶部到第一个 `---` 分隔线）。

每个新里程碑开始时，先做轻量意图校准：读最新快照中该里程碑的目标和备注（含 diff 标注），结合前序里程碑的实际完成现状，确认前提是否仍然成立、scope 是否需要调整。若无偏差则快速锁定进入执行；若标记为 🔁 需重评 或前提已变，则递归回 imp-intent 做完整意图循环。不盲目继续上次的计划，也不过度重复已确认的意图。若需要了解变更历史，往下读早期版本快照。

### Step 2: 任务拆解（3-7 个可独立验证单元）

基于锁定意图，拆解任务列表，输出供用户确认。

### Step 3: 皮骨可行性检查

- 需改数据模型？→ 标记 [需骨架评估]，升级至 imp-architect，退出此流程
- 骨架能支撑？→ 否则暂停
- 本次任务是否涉及长反馈循环（构建镜像、安装大型依赖、部署流水线等，单次试错 > 分钟级）？是 → 执行前先做轻量预检（版本兼容性查证、dry-run、配置校验等），把能提前发现的错误在动手前排除
- 通过 → 继续

### Step 4: 逐任务执行 + 逐任务调用 imp-verify

每个任务完成后立即验证，不积累到最后。

**执行中遇到需自主判断的选择时**：

- 若该选择会改变已确认意图的实现路径、涉及资源/成本取舍、或超出用户已知的上下文 → **必须暂停，先做简洁汇报**：
  ```
  [IMP-汇报] 自主判断: {做了什么选择}
  [IMP-汇报] 理由: {为什么这样选}
  [IMP-汇报] 需要你确认: 同意 / 建议换方案？
  ```
  等用户回复后再继续。若用户未明确反对，默认同意继续。
- 若选择完全在意图范围内且用户已明确偏好 → 可直接执行，事后在验证结果中提及即可

**原则**：不让用户感到被频繁打扰，也不让用户对关键判断一无所知。

### Step 5: 更新 milestone-state.md，写入 session-state

在 `.windsurf/memory/milestone-state.md` **文件顶部追加一个新版本快照**（格式同 imp-architect Step 4 的追加快照模式）：

- 复制上一版本的完整表格作为新版本基础
- 更新当前里程碑的状态和备注，**记录实际完成内容而非复制原计划**
- 若实际结果与原计划有偏差（计划做 A，实际做了 A' 或 A+B），在备注列标注 `diff: {偏差说明和原因}`
- 若本次完成改变了后续里程碑的前提假设，将受影响的里程碑状态改为 🔁 需重评，备注标注原因

更新 `.windsurf/memory/session-state.md`。

session-state 只写下次对话必须知道的最少信息，详略由 AI 自决：简单任务一行，复杂任务不超过5行。格式：任务 / 当前进展 / 下一步。
