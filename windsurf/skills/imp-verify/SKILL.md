# IMP Verify — 落地验证

## 作用

「落地三循环」的验证阶段：计划 → 落地 → **验证**。
每个闭合工作单元完成后必须执行。

## 验证分级

| 变更类型 | 最低验证要求 |
|----------|-------------|
| Bug 修复 | 原路径恢复正常 + 无新回归 |
| UI 微调 | 视觉效果确认（截图或描述） |
| 功能新增 | 用户入口可达 + 核心路径通过 |
| API 变更 | 接口返回符合预期 + 调用方兼容 |
| 骨架变更 | 全链路 Smoke + 数据完整性确认 |

## 验证两步法

1. AI 先基于已知上下文 reasoning 验证结论
2. 若不确定 → 主动扫描相关文件或请求运行测试

## 验证输出格式（强制，禁止只说「已完成」）

```
[IMP-Verify] 变更: {描述}
[IMP-Verify] 验证方式: {手工/Smoke/单元测试}
[IMP-Verify] 验证路径: {操作步骤或脚本名}
[IMP-Verify] 结果: {通过/失败} — {关键输出}
```

## 验证未通过时

输出：

```
[IMP-Verify] ❌ 失败原因: {描述}
```

**先对失败原因重新判级**（复用 Global Rules Step 1 的级别判定）：

- 问题在当前级别内可修（如任务级的小 bug）→ 回到对应 Skill 的执行阶段修复
- 问题超出当前级别（如在任务级/功能级验收时发现骨架设计错误）→ **升级路由**，走 imp-intent → imp-architect，不得在当前级别直接修改骨架
- 问题影响到其他里程碑的前提假设 → 更新 `milestone-state.md`，将受影响里程碑标注 `[需重评]`

不得跳过验证继续推进，不得在低级别直接修复高级别问题。
