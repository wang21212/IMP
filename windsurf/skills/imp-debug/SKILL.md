# IMP Debug — 任务级执行流程

## 适用范围

执行明确指令：修 bug、跑验收、小调整、格式化等。不新增能力，不改架构。

## 流程

### Step 1: 复现路径确认

描述触发条件，确认是否稳定复现。

### Step 2: 顺藤摸瓜定位

按调用链方向：用户操作 → 前端 → API → 业务逻辑 → 数据层

输出：

```
[IMP-Debug] 问题入口: {文件}:{行号} — {原因假设}
```

### Step 3: 皮骨检查（必须）

- 修复是否改数据模型？是 → 升级骨架级，退出此流程，重新路由至 imp-intent → imp-architect
- 修复是否改接口签名？是 → 升级功能级，退出此流程，重新路由至 imp-intent → imp-feature
- 本次操作是否属于长反馈循环（构建镜像、安装大型依赖、部署流水线等，单次试错 > 分钟级）？是 → 执行前必须先做轻量预检（版本兼容性查证、dry-run、配置校验等），把能提前发现的错误在动手前排除，避免「构建→失败→重建→再失败」的高代价循环
- 均否 → 继续

### Step 4: 修复 + 调用 imp-verify

实施最小改动，执行 imp-verify，输出验证结果。

### Step 5: 写入 session-state

更新 `.windsurf/memory/session-state.md`。

session-state 只写下次对话必须知道的最少信息，详略由 AI 自决：任务级通常一行即可，复杂排查不超过3行。格式：任务 / 执行结果 / 是否有遗留。
